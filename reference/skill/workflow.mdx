The prompt-improver skill follows a systematic 4-phase approach to transform vague prompts into actionable requests.

## Overview

When a prompt is determined to be vague (by the UserPromptSubmit hook), the skill guides you through:

<Steps>
  <Step title="Research">
    Gather context through conversation history, codebase exploration, and documentation
  </Step>
  
  <Step title="Generate Questions">
    Formulate targeted questions based on research findings
  </Step>
  
  <Step title="Get Clarification">
    Present questions to user with specific, grounded options
  </Step>
  
  <Step title="Execute">
    Fulfill the original request using clarifications and research
  </Step>
</Steps>

<Note>
The skill assumes the prompt has already been identified as vague by the hook. It proceeds directly to research and clarification.
</Note>

## Phase 1: Research

**Goal:** Gather context to ground your questions in actual findings rather than assumptions.

### Create Research Plan

Before executing research, use TodoWrite to create a dynamic research plan:

```markdown
Research Plan for "[Prompt]":
1. [What to research] - [Tool/approach]
2. [What to research] - [Tool/approach]
3. [What to research] - [Tool/approach]
```

**Example for "fix the bug":**

<CodeGroup>
```markdown Research Plan
Research Plan for "fix the bug":
1. Check conversation history for error messages
2. Search for failing tests - Grep for "failing", "error", "TODO"
3. Explore recent commits - Git log for problem areas
4. Identify patterns in similar code - Read related files
```

```bash Execution
# Step 1: Check conversation history
(Review recent messages for context)

# Step 2: Search for issues
Grep: "TODO|FIXME|BUG"
Glob: "**/*.test.ts"
Bash: npm test

# Step 3: Git history
Bash: git log --grep="fix|bug" --oneline -10

# Step 4: Read related files
Read: auth.ts, test_auth.test.ts
```
</CodeGroup>

### Research Priority Order

<Steps>
  <Step title="Check conversation history FIRST">
    Look for:
    - Recent error messages
    - File names mentioned
    - Features discussed
    - Decisions already made
    
    **Critical:** Conversation history can make vague prompts clear without codebase exploration.
  </Step>
  
  <Step title="Review codebase if needed">
    Use Task/Explore for architecture, Grep/Glob for specific patterns:
    - Task/Explore: Map project structure, find similar implementations
    - Grep: Search for function calls, error patterns, TODOs
    - Glob: Find files by type or naming pattern
    - Bash (git): Check recent changes, commit history
  </Step>
  
  <Step title="Gather additional context">
    When needed:
    - Read: Local documentation (README.md, docs/)
    - WebFetch: Official documentation for libraries
    - WebSearch: Best practices, common patterns, current approaches
  </Step>
  
  <Step title="Document findings">
    Summarize what you learned before asking questions:
    - Key files involved
    - Existing patterns found
    - Constraints discovered
    - Options identified from research
  </Step>
</Steps>

## Phase 2: Generate Targeted Questions

**Goal:** Formulate 1-6 questions that will clarify the ambiguity, with each option traceable to research.

### Question Guidelines

<CardGroup cols={2}>
  <Card title="Grounded" icon="anchor">
    Every option comes from research: codebase findings, documentation, or common patterns discovered
  </Card>
  
  <Card title="Specific" icon="bullseye">
    Avoid vague options like "Other approach". Use concrete choices like "JWT with HttpOnly cookies"
  </Card>
  
  <Card title="Multiple Choice" icon="list">
    Provide 2-4 concrete options per question with context about trade-offs
  </Card>
  
  <Card title="Focused" icon="filter">
    Each question addresses one decision point. Separate compound questions.
  </Card>
</CardGroup>

### Number of Questions

<Tabs>
  <Tab title="1-2 Questions">
    **Simple ambiguity**
    
    Use when:
    - Which file?
    - Which approach?
    - Single decision point
    
    **Example:** "fix the bug" → Which bug (after finding 3 potential bugs in research)
  </Tab>
  
  <Tab title="3-4 Questions">
    **Moderate complexity**
    
    Use when:
    - Scope + approach + validation
    - Multiple independent decisions
    - Configuration choices
    
    **Example:** "add validation" → Where to add? What approach? How to handle existing validation?
  </Tab>
  
  <Tab title="5-6 Questions">
    **Complex scenarios**
    
    Use when:
    - Major feature with architectural decisions
    - Multiple aspects needing clarification
    - Configuration + approach + scope + priority
    
    **Example:** "add authentication" → Auth method? Token storage? Scope (login/register/roles/verification)? Migration strategy?
    
    <Warning>
    Only use 5-6 questions when truly necessary. Most scenarios should use 1-4 questions.
    </Warning>
  </Tab>
</Tabs>

## Phase 3: Get Clarification

**Goal:** Present your research-grounded questions to the user and gather their decisions.

### Use AskUserQuestion Tool

<ParamField path="questions" type="array" required>
  Array of question objects. Each question must include:
  
  <Expandable title="Required fields">
    <ResponseField name="question" type="string" required>
      Clear, specific question ending with `?`
    </ResponseField>
    
    <ResponseField name="header" type="string" required>
      Short label for UI (max 12 characters)
    </ResponseField>
    
    <ResponseField name="multiSelect" type="boolean" required>
      - `false`: Single selection (most common)
      - `true`: Multiple selections allowed
    </ResponseField>
    
    <ResponseField name="options" type="array" required>
      2-4 specific choices, each with:
      - `label`: Concise choice (1-5 words)
      - `description`: Context, trade-offs, implications
    </ResponseField>
  </Expandable>
</ParamField>

### Important Guidelines

<Warning>
- Always include `multiSelect` field (true/false)
- User can always select "Other" for custom input (don't add generic "Other" option)
- Include brief explanations of trade-offs in descriptions
- Keep header ≤12 characters
</Warning>

## Phase 4: Execute with Context

**Goal:** Fulfill the original request as if it had been clear from the start.

### Execution Inputs

Proceed using:

<Steps>
  <Step title="Original prompt intent">
    What the user was trying to accomplish
  </Step>
  
  <Step title="Clarification answers">
    User's selections from your questions
  </Step>
  
  <Step title="Research findings">
    Context gathered during research phase
  </Step>
  
  <Step title="Conversation history">
    Additional context from recent messages
  </Step>
</Steps>

### Execution Example

**Original prompt:** "fix the bug"

**User answer:** "Token validation (auth.ts:89)"

**Research context:**
- FIXME comment at auth.ts:89
- 2 failing tests in test_auth.test.ts
- Error involves JWT token validation

**Execution:**

<CodeGroup>
```typescript Read auth.ts:89
// auth.ts:89 - FIXME: Token validation is broken
function validateToken(token: string) {
  // Current broken implementation
  return jwt.verify(token, process.env.SECRET);
}
```

```typescript Fix implementation
function validateToken(token: string) {
  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    
    // Check token expiration
    if (decoded.exp && decoded.exp < Date.now() / 1000) {
      throw new Error('Token expired');
    }
    
    return decoded;
  } catch (error) {
    if (error.name === 'JsonWebTokenError') {
      throw new Error('Invalid token');
    }
    throw error;
  }
}
```

```bash Verify fix
# Run the failing tests
npm test test_auth.test.ts

# Result: Tests now passing ✓
```
</CodeGroup>

<Tip>
Execute as if the prompt had been: "Fix the token validation bug in auth.ts:89 that's causing test failures"
</Tip>

## Workflow Checklist

Use this checklist for each vague prompt:

### Research Phase
- [ ] Created research plan with TodoWrite
- [ ] Checked conversation history first
- [ ] Explored codebase for patterns
- [ ] Gathered additional context (docs, web)
- [ ] Documented findings before asking questions

### Question Generation Phase
- [ ] Each option grounded in research findings
- [ ] 2-4 options per question
- [ ] 1-6 questions total based on complexity
- [ ] Each question focuses on one decision point
- [ ] Descriptions include trade-offs and context

### Clarification Phase
- [ ] Used AskUserQuestion tool
- [ ] All required fields present (question, header, multiSelect, options)
- [ ] Header ≤12 characters
- [ ] Question ends with `?`

### Execution Phase
- [ ] Used original prompt intent
- [ ] Applied user's clarification answers
- [ ] Incorporated research findings
- [ ] Considered conversation history
- [ ] Executed as if prompt was clear from start
