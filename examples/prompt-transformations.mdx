### Generic Bug Fix

This example shows the full journey from a completely vague prompt to precise execution.

<Tabs>
  <Tab title="Original Prompt">
    ```bash
    $ claude "fix the bug"
    ```

    **Evaluation:**
    - Target: ❌ (no file, function, or bug specified)
    - Action: ❌ (what needs fixing?)
    - Criteria: ~ (bug fixed, but which bug?)
    - Context: Check conversation history

    **Conversation History Check:**
    No recent error messages, no file viewing context.

    **Decision:** RESEARCH REQUIRED
  </Tab>

  <Tab title="Research Phase">
    Claude creates a research plan using TodoWrite:

    ```
    Research Plan:
    1. Search for TODO/FIXME comments indicating known bugs
    2. Check for failing tests
    3. Look for recent git commits mentioning "fix" or "bug"
    4. Search for try/catch blocks with console.error
    ```

    **Research Execution:**
    ```bash
    # Step 1: Search for known issues
    Grep: "TODO|FIXME|BUG"
    Results: 3 TODOs, 1 FIXME in auth.ts:89

    # Step 2: Check test failures
    Glob: "**/*.test.ts"
    Bash: npm test
    Results: 2 tests failing in test_auth.test.ts

    # Step 3: Review recent fixes
    Bash: git log --grep="fix|bug" --oneline -10
    Results: Recent commit "fix login redirect" 2 days ago

    # Step 4: Find error logging
    Grep: "console.error|console.warn"
    Results: Error logged in auth.ts:145 "Invalid token"
    ```

    **Research Findings:**
    - FIXME in auth.ts:89: "FIXME: Token validation is broken"
    - 2 failing tests related to token validation
    - Recent fix attempt for login redirect
    - Error logging at auth.ts:145 for invalid tokens
  </Tab>

  <Tab title="Questions Asked">
    Based on research findings, Claude asks:

    ```json
    {
      "questions": [
        {
          "question": "Which bug should be fixed?",
          "header": "Bug target",
          "multiSelect": false,
          "options": [
            {
              "label": "Token validation (auth.ts:89)",
              "description": "FIXME comment indicates broken token validation. 2 failing tests in test_auth.test.ts related to this. Likely high priority."
            },
            {
              "label": "Login redirect (recent commit)",
              "description": "Recent commit 2 days ago attempted to fix login redirect. May have residual issues."
            },
            {
              "label": "Invalid token logging (auth.ts:145)",
              "description": "Errors being logged for invalid tokens. May be symptom of validation bug or separate issue."
            }
          ]
        }
      ]
    }
    ```

    **User Answer:** "Token validation (auth.ts:89)"
  </Tab>

  <Tab title="Execution">
    With clarification, Claude proceeds with full context:

    ```typescript
    // Fix token validation in auth.ts:89
    // Addresses FIXME comment and resolves 2 failing tests

    // Before (auth.ts:89)
    function validateToken(token: string) {
      // FIXME: Token validation is broken
      return jwt.verify(token, process.env.SECRET);
    }

    // After (auth.ts:89)
    function validateToken(token: string) {
      try {
        const decoded = jwt.verify(token, process.env.JWT_SECRET);
        if (!decoded || typeof decoded !== 'object') {
          throw new Error('Invalid token structure');
        }
        return decoded;
      } catch (error) {
        console.error('Token validation failed:', error.message);
        throw new Error('Invalid token');
      }
    }
    ```

    **Verification:**
    ```bash
    npm test test_auth.test.ts
    # ✅ All tests passing
    ```
  </Tab>
</Tabs>

    ```typescript
    // src/auth/jwt.ts
    import jwt from 'jsonwebtoken';

    export function generateAccessToken(userId: string, role: string) {
      return jwt.sign({ userId, role }, process.env.JWT_SECRET, {
        expiresIn: '15m'
      });
    }

    export function generateRefreshToken(userId: string) {
      return jwt.sign({ userId }, process.env.REFRESH_SECRET, {
        expiresIn: '7d'
      });
    }
    ```

    ```typescript
    // src/routes/auth.ts
    router.post('/register', async (req, res) => {
      const { email, password, role } = req.body;
      const user = await User.create({ email, password, role: role || 'user' });
      
      const accessToken = generateAccessToken(user.id, user.role);
      const refreshToken = generateRefreshToken(user.id);
      
      // HttpOnly cookie for web
      res.cookie('refreshToken', refreshToken, { httpOnly: true, secure: true });
      
      // Also return in body for mobile
      res.json({ accessToken, refreshToken });
    });

    router.post('/login', async (req, res) => {
      // Similar dual-mode response
    });
    ```

    ```typescript
    // src/middleware/auth.ts
    export function requireAuth(req, res, next) {
      // Check both Authorization header and cookie
      const token = req.headers.authorization?.split(' ')[1] || req.cookies.accessToken;
      
      if (!token) return res.status(401).json({ error: 'Unauthorized' });
      
      try {
        const decoded = jwt.verify(token, process.env.JWT_SECRET);
        req.user = decoded;
        next();
      } catch (error) {
        res.status(401).json({ error: 'Invalid token' });
      }
    }

    export function requireRole(role: string) {
      return (req, res, next) => {
        if (req.user.role !== role) {
          return res.status(403).json({ error: 'Forbidden' });
        }
        next();
      };
    }
    ```

## Key Transformation Patterns

<AccordionGroup>
  <Accordion title="Generic to Specific">
    **Pattern:** Vague action words ("fix", "improve", "refactor") are transformed into precise targets through codebase research.

    **Example transformations:**
    - "fix the bug" → "Fix token validation in auth.ts:89 (FIXME comment, 2 failing tests)"
    - "refactor the code" → "Extract UserService class following ProductService pattern"
    - "improve performance" → "Add Redis caching to /api/products with 1-minute TTL"
  </Accordion>

  <Accordion title="Feature to Architecture">
    **Pattern:** Broad feature requests become architectural decisions through stack analysis and best practices research.

    **Example transformations:**
    - "add authentication" → "JWT with refresh tokens, dual-mode (cookies + headers), with RBAC"
    - "implement caching" → "Redis with 1-minute TTL for product catalog and user profiles"
    - "add logging" → "Winston with rotating file transport, JSON format for log aggregation"
  </Accordion>

  <Accordion title="Ambiguous to Multi-Faceted">
    **Pattern:** Single vague requests expand into multiple decision points when research reveals complexity.

    **Example transformations:**
    - "implement caching" → 3 questions: target endpoints, strategy (Redis vs in-memory), TTL duration
    - "add authentication" → 3 questions: auth method, token storage, scope (login/register/reset/RBAC)
    - "improve error handling" → 2 questions: logging strategy, error response format
  </Accordion>
</AccordionGroup>

<Note>
All options presented to users are grounded in actual research findings from the codebase, documentation, or best practices—never generic assumptions.
</Note>
