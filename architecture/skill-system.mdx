## Overview

The skill layer provides systematic research and question generation guidance when the hook determines a prompt is vague. It uses progressive disclosure to load detailed references only when needed.

<Info>
**Location**: `skills/prompt-improver/` with core workflow in SKILL.md (~170 lines) and on-demand reference files (700-1000 lines total).
</Info>

## Skill Invocation

The skill can be invoked in two ways:

### Automatic Invocation (Primary)

```markdown
1. User submits prompt: "fix the bug"
2. Hook wraps with evaluation prompt
3. Claude evaluates: vague (no target, no context)
4. Claude invokes: prompt-improver skill
5. Skill guides through research and questioning
```

### Manual Invocation (Secondary)

```markdown
Use the prompt-improver skill to research and clarify: "add authentication"
```

<Tip>
Manual invocation is useful for testing, development, or when working on prompt evaluation systems.
</Tip>

## 4-Phase Workflow

The skill guides Claude through a systematic process:

### Phase 1: Research

**Purpose**: Gather context before formulating questions

**Process**:

1. **Create research plan using TodoWrite**
   ```markdown
   Research Plan:
   1. Check conversation history first
   2. Review codebase if needed
   3. Gather additional context as needed
   4. Document findings
   ```

2. **Check conversation history first**  
   Avoid redundant exploration if context already exists

3. **Review codebase if needed**:
   - Task/Explore for architecture and project structure
   - Grep/Glob for specific patterns, related files
   - Check git log for recent changes
   - Search for errors, failing tests, TODO/FIXME comments

4. **Gather additional context**:
   - Read local documentation files
   - WebFetch for online documentation
   - WebSearch for best practices, common approaches

5. **Document findings**  
   Ground questions in actual project context

**Critical Rules** (from SKILL.md:50-54):
- NEVER skip research
- Check conversation history before exploring codebase
- Questions must be grounded in actual findings, not assumptions

<Warning>
Skipping research leads to generic questions based on base knowledge rather than project-specific context.
</Warning>

### Phase 2: Generate Targeted Questions

**Purpose**: Formulate 1-6 questions based on research findings

**Question Guidelines** (from SKILL.md:58-72):

<AccordionGroup>
  <Accordion title="Grounded">
    Every option comes from research:
    - Codebase findings
    - Documentation references
    - Common patterns from web search
    - Git history patterns
    
    **Bad**: "Use some state management"  
    **Good**: "Use Redux (existing pattern in src/store/)" ← grounded in codebase
  </Accordion>

  <Accordion title="Specific">
    Avoid vague options:
    
    **Bad**: "Use a different approach"  
    **Good**: "Use JWT tokens with HttpOnly cookies"
  </Accordion>

  <Accordion title="Multiple Choice">
    Provide 2-4 concrete options per question:
    
    ```markdown
    Which authentication approach?
    ○ JWT tokens with HttpOnly cookies (existing pattern)
    ○ Session-based with Redis (requires new dependency)
    ○ OAuth 2.0 with third-party provider
    ```
  </Accordion>

  <Accordion title="Focused">
    One decision point per question:
    
    **Bad**: "Which file and what approach?" (two questions)
    **Good**: "Which file?" + separate "What approach?" question
  </Accordion>

  <Accordion title="Contextual">
    Include brief explanations of trade-offs:
    
    ```markdown
    ○ JWT tokens (stateless, scales horizontally)
    ○ Sessions (simpler, requires state management)
    ```
  </Accordion>
</AccordionGroup>

**Number of Questions** (from SKILL.md:67-72):
- **1-2 questions**: Simple ambiguity (which file? which approach?)
- **3-4 questions**: Moderate complexity (scope + approach + validation)
- **5-6 questions**: Complex scenarios (major feature with multiple decision points)

### Phase 3: Get Clarification

**Purpose**: Present research-grounded questions using AskUserQuestion tool

**Tool Format** (from SKILL.md:76-90):

<ParamField path="question" type="string" required>
  Clear, specific question ending with ?
</ParamField>

<ParamField path="header" type="string" required>
  Short label (max 12 chars) for UI display
</ParamField>

<ParamField path="multiSelect" type="boolean" required>
  false unless choices aren't mutually exclusive. User can always select "Other" for custom input.
</ParamField>

<ParamField path="options" type="array" required>
  Array of 2-4 specific choices from research
  
  <ParamField path="label" type="string" required>
    Concise choice text (1-5 words)
  </ParamField>
  
  <ParamField path="description" type="string">
    Context about this option (trade-offs, implications)
  </ParamField>
</ParamField>

**Example**:

```json
{
  "question": "Which bug needs fixing?",
  "header": "Target",
  "multiSelect": false,
  "options": [
    {
      "label": "Login auth failure",
      "description": "auth.py:145 - try/catch swallowing errors"
    },
    {
      "label": "Session timeout",
      "description": "session.py:89 - premature expiration"
    },
    {
      "label": "Other",
      "description": "Paste error message"
    }
  ]
}
```

### Phase 4: Execute with Context

**Purpose**: Proceed with original request using clarifications

**Available Context** (from SKILL.md:92-99):
- Original prompt intent
- Clarification answers from user
- Research findings and context
- Conversation history

**Execution Principle**:  
Execute the request as if it had been clear from the start.

## Key Principles

From SKILL.md:144-150:

1. **Assume Vagueness**: Skill is only invoked for vague prompts (evaluation done by hook)
2. **Research First**: Always gather context before formulating questions
3. **Ground Questions**: Use research findings, not assumptions or base knowledge
4. **Be Specific**: Provide concrete options from actual codebase/context
5. **Stay Focused**: Max 1-6 questions, each addressing one decision point
6. **Systematic Approach**: Follow 4-phase workflow (Research → Questions → Clarify → Execute)
